Concepts

How it works

testRigor runs UI tests written in plain English on cloud browsers. Claude Code is the agent that drives them. These skills are the glue: your tests live as files in your repo, the testrigor CLI pushes them to a cloud test suite and runs them, and results come back as an exit code plus a JUnit report. You can point a run at a deployed URL — or at localhost, reached through testRigor's tunnel.

1
BuildWrite test cases & reusable rules as plain-English files in your repo.
2
Run on localhostThe CLI tunnels testRigor's cloud browsers to your local app.
3
DebugRead the exit code, the JUnit report, and the run URL; fix; repeat.
4
Update remoteRe-running pushes your files, so the cloud suite always matches your repo.
Plain English, not selectors

Refer to what the user sees — click "Sign in" — never CSS/XPath.

Tests as code

Test cases, rules and variables are files you version alongside your app.

Environment-agnostic

The same tests run on local, staging, or prod — the URL is supplied at run time.

The skills

Four focused skills

Each folder is a self-contained Claude Code skill (a SKILL.md plus optional references). They're split so Claude loads only what's relevant to the moment — authoring, running, API steps, or the full loop.

Install

Set it up in a couple of minutes

You need a testRigor account (for an API token), Node.js 18+, and Claude Code.

Install the testRigor CLI

The skills drive this binary (testrigor).

# Node.js 18+
npm install -g testrigor-cli
testrigor --version

Authenticate (non-interactive)

Create a Personal Authentication Token in testRigor (your name → API Tokens → Generate). Export it so Claude/CI never hit an interactive prompt.

export TESTRIGOR_API_KEY="your-personal-auth-token"

Install the skills into Claude Code

Copy the skill folders into Claude Code's skills directory.

git clone https://github.com/TestRigor/skills.git

# Personal (available in every project):
mkdir -p ~/.claude/skills
cp -r skills/testrigor-* ~/.claude/skills/

# …or project-scoped (commit them with one repo):
# cp -r skills/testrigor-* <your-project>/.claude/skills/
Claude Code discovers any SKILL.md under ~/.claude/skills/ (personal) or <project>/.claude/skills/ (project). Each testrigor-* folder is one skill — drop in all four.

Usage

Using it with Claude

Point the CLI at a test suite, then just ask Claude — it pulls in the right skill automatically based on what you're doing.

1 · Point at a suite

Grab the suite ID from its URL in the testRigor app.

testrigor test-suite config --default <YOUR_SUITE_ID>

2 · Ask Claude

Write a testRigor test that logs in and checks the dashboard, then run it against http://localhost:3000.
My checkout test is failing — debug it from the JUnit report and fix it.
Add API seeding to this test with call api, then verify the result in the UI.

3 · The loop Claude runs for you

Serve your app locally; the tests run against it through the tunnel and the suite is updated by the same command.

# your app on :3000, then:
testrigor test-suite run --localhost --url "http://localhost:3000" \
  --test-cases-path "tests/**/*.txt" \
  --rules-path "rules/**/*.txt" \
  --explicit-mutations \
  --junit-report-save-path report.xml

Exit code 0 and failures="0" in report.xml mean green. For the full walkthrough, open testrigor-dev-loop or run an example.
